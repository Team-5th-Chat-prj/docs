# 11. 백엔드 패키지 구조 설계서

> **버전**: v2.1 (location 도메인 추가 / global/client/kakao 추가 / NearbyProductController 반영 / Lettuce 분산락 및 비관적 락 병용 반영)
> **패키지 루트**: `com.clone.getchu`
> **설계 원칙**: 도메인형(Domain-Driven) + 계층형(Layered) 혼합 구조
>
> **핵심 설계 결정 요약**
>
> | 항목 | 결정 | 이유 |
> |------|------|------|
> | 구조 방식 | 도메인형 최상위 + 계층형 내부 | 도메인 간 결합도 최소화, 기능 집중성 확보 |
> | 카테고리 구조 | 단층(depth 없음) | 대분류/소분류 계층 없이 단일 카테고리 목록으로 단순화 |
> | 락 전용 Facade 분리 | `TradeFacade`, `LikeFacade` | `@Transactional` ↔ Lettuce SETNX Lock 생명주기 충돌 방지 |
| 카카오 API 클라이언트 위치 | `global/client/kakao` | 외부 API 호출은 도메인이 아닌 전역 인프라로 분류 |
| location 도메인 분리 | `domain/location` (독립 패키지) | 동네 인증 로직(외부 API 호출 + DB 저장 분리)을 member에서 독립 |
> | QueryDSL Custom 분리 | `ProductRepositoryCustom` | 동적 쿼리를 JPA Repository와 분리하여 관심사 명확화 |
> | WebSocket Config 위치 | `global/config` | 인프라 설정은 도메인이 아닌 전역 관심사로 분류 |
> | Like 도메인 분리 | `domain/like` (독립 패키지) | Product 패키지 비대화 방지, 찜/찜취소 동시성 제어 독립 관리 |
> | Search 도메인 분리 | `domain/search` (독립 패키지) | 인기 검색어(Redis) 로직을 Product에서 분리 |

## 1. 전체 패키지 트리

```
src/main/java/com/clone/getchu/
│
├── GetchuApplication.java             # Spring Boot 메인 클래스
│
├── global/                             # 앱 전체 공통 관심사

```
domain/product/
├── controller/

│   │                                  # GET  /products (목록/검색 통합)
│   │                                  # GET  /products/{id}

│   │                                  # location(POINT SRID 4326), locationName — 동네 인증 위치

├── repository/
│   ├── ProductRepository.java         # JpaRepository — 기본 CRUD + 근처 상품 Native Query
│   │                                  # getNearbyProducts() — ST_Distance_Sphere 반경 조회

### 3-6. `trade` — 거래 · 동시성 제어 ⭐⭐

동시성 제어(Lettuce 분산락 + 비관적 락)와 상태 전이 규칙이 집중된 핵심 도메인.

```
domain/trade/
├── controller/
│   └── TradeController.java           # POST /products/{id}/reserve
│                                      # PATCH /trades/{id}/status
│                                      # GET   /trades/{id}
│                                      # GET   /members/me/trades?role={SELLER|BUYER}
├── dto/
│   ├── request/
│   │   └── TradeStatusUpdateRequest.java  # { status: TradeStatus }
│   └── response/
│       ├── TradeReserveResponse.java  # { tradeId, productTitle, sellerNickname, buyerNickname }
│       ├── GetTradeDetailResponse.java# { productTitle, status, price, productCreatedAt, soldAt, sellerNickname, buyerNickname }
│       └── GetAllTradeResponse.java   # { tradeId, productTitle, productId, price, status, updatedAt }
├── entity/
│   └── Trade.java                     # JPA 엔티티 (product FK, buyer FK, seller FK, status)
│                                      # proceed(role) / cancel(role) — 상태 전이 메서드
│                                      # reservedAt, tradedAt, soldAt — 각 단계 타임스탬프
├── enums/
│   ├── TradeStatus.java               # SALE → RESERVED → TRADING → SOLD → REVIEWED
│   │                                  # next(role) / cancel(role) — 전이 규칙 내장
│   └── TradeRole.java                 # SELLER | BUYER
├── repository/
│   └── TradeRepository.java           # JpaRepository
│                                      # existsByProductIdAndStatusIn() — 활성 거래 중복 확인
└── service/
    ├── TradeFacade.java               # ⭐ Lettuce SETNX 분산락 전용 Facade (예약 시)
    └── TradeService.java              # @Transactional — 예약/상태변경/조회
                                       # PRODUCT.status ↔ TRADE.status 동기화 (단일 트랜잭션)
```

> **Facade 분리 이유 (면접 포인트)**:
>
> ```
> [잘못된 구조]                        [올바른 구조]
> TradeService {                       TradeFacade {           ← 락 담당 (트랜잭션 없음)
>   @Transactional                       lock.tryLock()
>   reserve() {                          tradeService.reserve()    ← 내부에서 @Transactional
>     lock.tryLock()                     lock.unlock()             ← COMMIT 이후 해제 보장
>     ... DB 처리 ...               }
>     lock.unlock()  ← 문제!         TradeService {           ← DB 처리 담당
>   }                                  @Transactional
> }                                    reserve() { ... }
>                                    }
>
> 문제: COMMIT 전에 unlock → 다른 스레드가 아직 반영 전 DB 상태를 읽음
> 해결: Facade가 트랜잭션 외부에서 lock/unlock 생명주기를 감싸도록 분리
> ```

### 3-7. `chat` — 실시간 채팅

> **WebSocket 설정 위치**: `global/config/WebSocketConfig.java`
> - STOMP 브로커 엔드포인트(`/ws-chat`), SockJS 폴백 등록
> - JWT 검증: `global/security/StompChannelInterceptor.java`에서 STOMP CONNECT 시 1회 검증
>
> **메시지 저장 전략**: Redis Pub/Sub이 아닌 MySQL 직접 저장
> - 재연결 시 REST API(`GET /chat-rooms/{id}/messages`)로 이전 메시지 재조회 가능
> - Cursor 기준: `messageId DESC`

### 3-9. `location` — 동네 인증 ⭐

외부 API(카카오) 호출 + DB 저장을 분리한 설계. `LocationService`는 카카오 API 호출(트랜잭션 없음), `LocationUpdater`는 DB 저장(짧은 트랜잭션)으로 분리하여 외부 API 대기 중 DB 커넥션 점유 방지.

```
domain/location/
├── controller/
│   └── LocationController.java        # POST /api/location/verify/gps
│                                      # POST /api/location/verify/address
├── dto/
│   ├── request/
│   │   ├── GpsVerifyRequest.java      # { lat(-90~90), lng(-180~180) } + @Valid
│   │   └── AddressVerifyRequest.java  # { address(max 200) } + @Valid
│   └── response/
│       └── LocationVerifyResponse.java # { locationName, locationRadius }
└── service/
    ├── LocationService.java           # 카카오 API 호출 (트랜잭션 없음)
    │                                  # verifyByGps(): coordToRegionName → DB 저장
    │                                  # verifyByAddress(): addressToCoord → coordToRegionName → DB 저장
    │                                  # JTS GeometryFactory(SRID 4326)로 Point 객체 생성
    └── LocationUpdater.java           # @Transactional — Member.location · locationName 저장
                                       # 트랜잭션을 최소 범위로 제한
```

> **위치 저장**: `Member.location` = JTS `Point(lng, lat)`, SRID 4326.
> **카카오 API 클라이언트**: `global/client/kakao/KakaoLocalApiClient` 주입.

## 5. 주요 Bean 의존 관계

```
[요청 처리 흐름 — 예약 API]

HTTP POST /products/{id}/reserve
  └─▶ TradeController
        └─▶ TradeFacade              ← Lettuce Lock 획득 (SETNX 스핀락)
              └─▶ TradeService        ← @Transactional 시작
                    ├─▶ ProductRepository — findByIdWithLock (비관적 락 병용)
                    ├─▶ Product.updateStatus(RESERVED)
                    ├─▶ Trade 생성 + 저장
                    └─▶ @Transactional COMMIT
              └─▶ TradeFacade         ← finally: lock.unlock() (Lua Script)

[찜하기 흐름 — 동시성 제어]

HTTP POST /products/{id}/likes
  └─▶ LikeController
        └─▶ LikeFacade               ← Lettuce Lock 획득 (product 단위)
              └─▶ LikeService         ← @Transactional
                    ├─▶ LikeRepository.findByProductIdAndMemberIdIncludingDeleted()
                    ├─▶ Like 생성 or restore()
                    └─▶ product.incrementLikeCount() ← 객체 수정 (Dirty Checking)
              └─▶ LikeFacade          ← finally: lock.unlock()

[인기 검색어 캐시 흐름]

GET /search/popular
  └─▶ SearchController
        └─▶ SearchService
              └─▶ @Cacheable("popularKeywords")
                    └─▶ Redis ZREVRANGE(key, 0, 9)  ← TOP 10

상품 검색 시 (GET /products?keyword=아이폰)
  └─▶ ProductController → ProductService
        └─▶ SearchService.incrementKeywordScore("아이폰")
              └─▶ Redis ZINCRBY(key, 1, "아이폰")  ← 점수 +1
```
