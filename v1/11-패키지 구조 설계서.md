# 11. 백엔드 패키지 구조 설계서

> **버전**: v1.0
> **패키지 루트**: `com.clone.carrot`
> **설계 원칙**: 도메인형(Domain-Driven) + 계층형(Layered) 혼합 구조
>
> **핵심 설계 결정 요약**
>
> | 항목 | 결정 | 이유 |
> |------|------|------|
> | 구조 방식 | 도메인형 최상위 + 계층형 내부 | 도메인 간 결합도 최소화, 기능 집중성 확보 |
> | 카테고리 구조 | 단층(depth 없음) | 대분류/소분류 계층 없이 단일 카테고리 목록으로 단순화 |
> | 락 전용 Facade 분리 | `TradeLockFacade` | `@Transactional` ↔ Redisson Lock 생명주기 충돌 방지 |
> | QueryDSL Custom 분리 | `ProductRepositoryCustom` | 동적 쿼리를 JPA Repository와 분리하여 관심사 명확화 |
> | WebSocket Config 위치 | `global/config` | 인프라 설정은 도메인이 아닌 전역 관심사로 분류 |

### 3-4. `product` — 매물 ⭐

복잡한 동적 검색(QueryDSL)과 캐싱(Redis Cache-Aside, Caffeine)이 집중된 도메인.

```
domain/product/
├── controller/
│   ├── ProductController.java      # GET /products · /products/{id} · POST · PATCH · DELETE
│   ├── ProductSearchController.java# GET /products/search · /search/popular
│   └── LikeController.java         # POST · DELETE /products/{id}/likes
│                                   # GET /members/me/likes
├── dto/
│   ├── request/
│   │   ├── CreateProductReq.java   # { title, description, price, categoryId, imageUrls[] }
│   │   ├── UpdateProductReq.java   # 부분 수정 — null 필드는 변경 없음
│   │   └── ProductSearchReq.java   # { keyword, categoryId, status, cursor, size }
│   └── response/
│       ├── ProductSummaryRes.java  # 목록용 { id, title, price, thumbnailUrl, status, likeCount }
│       ├── ProductDetailRes.java   # 상세용 + seller 정보 + imageUrls[]
│       └── PopularKeywordRes.java  # { keywords[], cachedAt }
├── entity/
│   ├── Product.java                # 매물 엔티티 (status, likeCount 비정규화 컬럼 포함)
│   ├── ProductImage.java           # PRODUCT_IMAGE (image_url, sort_order)
│   ├── Like.java                   # LIKE (member_id, product_id UNIQUE)
│   └── KeywordLog.java             # KEYWORD_LOG (keyword, count, log_date)
├── repository/
│   ├── ProductRepository.java          # JpaRepository — 기본 CRUD
│   ├── ProductRepositoryCustom.java    # [Interface] QueryDSL 동적 쿼리 계약
│   ├── ProductRepositoryImpl.java      # [Class] BooleanBuilder 기반 동적 검색 구현
│   │                                   #   + Cursor 페이지네이션 (createdAt DESC, id DESC)
│   ├── ProductImageRepository.java     # JpaRepository<ProductImage, Long>
│   ├── LikeRepository.java             # JpaRepository<Like, Long>
│   │                                   #   existsByMemberIdAndProductId() 포함
│   └── KeywordLogRepository.java       # 최근 7일 TOP 10 집계 네이티브 쿼리
└── service/
    ├── ProductService.java         # 매물 등록 · 수정 · 삭제 + Redis Cache-Aside (ADR-008)
    │                               #   수정·삭제 시 redisTemplate.delete("product:{id}") eviction
    ├── ProductSearchService.java   # 검색 + 검색어 로그 비동기 저장
    │                               #   @Cacheable("popularKeywords") → Caffeine (ADR-003)
    ├── PopularKeywordScheduler.java# @Scheduled(fixedRate=600000) — KEYWORD_LOG 집계 후 캐시 PUT
    └── LikeService.java            # 좋아요/취소 + like_count ±1 (허용 오차 비정규화)
```

> **QueryDSL Custom 패턴**:
> `ProductRepository`가 `ProductRepositoryCustom`을 상속하면, Spring Data가 `ProductRepositoryImpl`을 자동으로 주입한다. 복잡한 동적 쿼리와 JPA 기본 CRUD가 단일 Repository 인터페이스로 노출됨.

### 3-5. `trade` — 거래 · 동시성 제어 ⭐⭐

동시성 제어(Redisson + 비관적 락)와 상태 전이 규칙이 집중된 핵심 도메인.

```
domain/trade/
├── controller/
│   └── TradeController.java        # POST /products/{id}/reserve
│                                   # PATCH /trades/{id}/status
│                                   # GET   /trades/{id}
├── dto/
│   ├── request/
│   │   └── TradeStatusUpdateReq.java  # { status } — "TRADING" | "SOLD" | "SALE"(취소)
│   └── response/
│       └── TradeRes.java           # { tradeId, productId, status, chatRoomId }
├── entity/
│   └── Trade.java                  # JPA 엔티티 (product_id FK, buyer_id, seller_id, status)
│                                   #   정적 팩토리: Trade.create(product, buyerId)
├── repository/
│   └── TradeRepository.java        # JpaRepository
│                                   #   existsByProductIdAndStatusIn() — 활성 거래 중복 확인
│                                   #   existsByProductIdAndBuyerIdAndStatusNot() — 동일 구매자 중복
└── service/
    ├── TradeLockFacade.java        # ⭐ Redisson 분산락 전용 Facade (@Component, 트랜잭션 없음)
    │                               #   tryLock(wait=0, lease=3s) → reserveWithPessimisticLock 호출
    │                               #   Redis 장애 시 fallback: 비관적 락 단독 처리
    ├── TradeService.java           # [Interface]
    └── TradeServiceImpl.java       # [Class] @Transactional
                                    #   reserveWithPessimisticLock — SELECT FOR UPDATE + 상태 검증
                                    #   changeStatus — RESERVED→TRADING→SOLD / 취소 상태 전이 검증
                                    #   PRODUCT.status ↔ TRADE.status 동기화 (단일 트랜잭션)
```

> **Facade 분리 이유 (면접 포인트)**:
>
> ```
> [잘못된 구조]                        [올바른 구조]
> TradeService {                       TradeLockFacade {           ← 락 담당 (트랜잭션 없음)
>   @Transactional                       lock.tryLock()
>   reserve() {                          tradeService.reserve()    ← 내부에서 @Transactional
>     lock.tryLock()                     lock.unlock()             ← COMMIT 이후 해제 보장
>     ... DB 처리 ...               }
>     lock.unlock()  ← 문제!         TradeServiceImpl {           ← DB 처리 담당
>   }                                  @Transactional
> }                                    reserve() { ... }
>                                    }
>
> 문제: COMMIT 전에 unlock → 다른 스레드가 아직 반영 전 DB 상태를 읽음
> 해결: Facade가 트랜잭션 외부에서 lock/unlock 생명주기를 감싸도록 분리
> ```

## 5. 주요 Bean 의존 관계

```
[요청 처리 흐름 — 예약 API]

HTTP POST /products/{id}/reserve
  └─▶ TradeController
        └─▶ TradeLockFacade          ← Redisson Lock 획득 (wait=0, lease=3s)
              └─▶ TradeServiceImpl   ← @Transactional 시작
                    ├─▶ ProductRepository.findByIdWithLock()   ← SELECT FOR UPDATE
                    ├─▶ Product.changeStatus(RESERVED)
                    ├─▶ TradeRepository.save(trade)
                    └─▶ @Transactional COMMIT
              └─▶ TradeLockFacade    ← finally: lock.unlock()  (COMMIT 이후)

[인기 검색어 캐시 흐름]

@Scheduled(10분)
  └─▶ PopularKeywordScheduler
        ├─▶ KeywordLogRepository     ← 최근 7일 TOP 10 집계
        └─▶ CacheManager.put("popularKeywords", top10)   ← Caffeine 갱신

GET /search/popular
  └─▶ ProductSearchController
        └─▶ ProductSearchService
              └─▶ @Cacheable("popularKeywords")           ← HIT: 즉시 반환 / MISS: DB 집계 후 PUT

[상품 상세 캐시 흐름]

GET /products/{id}
  └─▶ ProductService.getProduct()
        ├─▶ redisTemplate.get("product:{id}")  ← HIT: 즉시 반환
        └─▶ ProductRepository.findById()       ← MISS: DB 조회 후 Redis PUT (TTL 10분)

PATCH /products/{id}
  └─▶ ProductService.updateProduct()
        └─▶ redisTemplate.delete("product:{id}")          ← Cache Eviction
```

## 6. 개발 우선순위 (3주 일정 기준)

| 주차 | 구현 대상 패키지 |
|------|----------------|
| 0주 (설계) | `global` 전체 뼈대, Entity · Repository 선언 |
| 1주 (개발1) | `auth`, `member`, `product` (기본 CRUD + 검색), `category` |
| 2주 (개발2) | `trade` (동시성 제어 포함), `chat` (WebSocket), 캐싱 · 좋아요 |
| 3주 (QA) | `review`, EXPLAIN 인덱스 최적화, 통합 테스트 |

> **공통**: ERD 기반 Entity는 0주에 전원이 함께 작성 후 도메인별 Repository · Service를 병렬 구현.
> Facade · Custom Repository 등 설계 패턴은 0주에 뼈대 파일(빈 클래스)로 미리 생성하여 팀원 간 구조를 공유한다.
