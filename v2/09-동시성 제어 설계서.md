> **버전**: v1.4 (Lettuce 분산락 + 비관적 락(Pessimistic Lock) 병용 전략 반영)
> **대상 기능**: 
> 1. 상품 예약 요청 (1명만 예약 가능)
> 2. 상품 찜하기/취소 (동시 요청 시 likeCount 정합성 보장)
>
> **예약 모델 정책**: 구매자가 [예약하기] 클릭 시 **판매자 승인 없이 즉시 RESERVED**
> **락 대상**: `PRODUCT` 테이블 — 상품 상태(SALE→RESERVED) 및 찜 수(likeCount) 변경. Redis 분산락(Lettuce) 및 DB 비관적 락 활용.

---

## 1. 문제 정의

### 상황

```
T=0ms  이서연: POST /products/42/reserve  ──┐
T=1ms  최우진: POST /products/42/reserve  ──┤──▶ 동시 도달
T=2ms  박준서: POST /products/42/reserve  ──┘

기대: 이서연만 성공, 최우진·박준서는 즉시 실패 또는 약간의 대기 후 실패
위험: 3명 모두 product.status = "SALE" 읽고 모두 RESERVED로 변경 시도
결과(제어 없을 때): 3명 모두 예약 성공 트랜잭션을 날려 데이터 정합성 파괴 (Lost Update 문제)
```

또는 여러 사용자가 동시에 같은 상품에 **찜하기(Like)** 버튼을 누르는 경우, DB에서 조회된 `likeCount`가 서로 동일한 상태에서 증가시키게 되어 최종 숫자가 누락되는 동시성 문제가 발생합니다.

### 왜 단순 `if (status == SALE)` 체크로 충분하지 않은가?

```
Thread1: SELECT status → SALE  (읽기)
Thread2: SELECT status → SALE  (읽기, Thread1이 아직 UPDATE 전)
Thread1: UPDATE status = RESERVED  (쓰기)
Thread2: UPDATE status = RESERVED  (쓰기, 이미 RESERVED인데도 덮어씀)
→ 로직상 검증을 통과해 DB에 2개의 예약(Trade) 레코드가 생성되는 치명적 버그!
```

## 2. 동시성 제어 전략 비교

| 전략 | 메커니즘 | 장점 | 단점 | 현재 프로젝트 적용 여부 |
|------|----------|------|------|-------------|
| **낙관적 락** | @Version 필드 추가 | 충돌 적을 때 성능 우수, DB 부하 적음 | 충돌 발생 시 클라이언트 단 재시도 로직 필요 | 미적용 |
| **Lettuce 분산락 (Redis)** | Redis SETNX + Lua Script | DB 부하 최소화, 분산 환경 완벽 대응 | Redis 네트워크 비용, 구현(스핀락) 복잡 | ✅ **최종 채택 (Trade, Like)** |
| **비관적 락 (Fallback)** | SELECT FOR UPDATE | 트랜잭션 충돌 원천 차단, 최종 정합성 보장 | DB 대기 발생 가능성 | ✅ **병용 채택 (Trade)** |

**최종 전략**: **Lettuce 분산락 + 비관적 락 병용 (Dual Lock)**
- **이유**: Lettuce 분산락으로 애플리케이션 레벨에서 1차 차단하여 DB 부하를 줄이고, `TradeService` 내부에서 `findByIdWithLock`을 통해 DB 레벨에서도 2차 방어하여 분산락 해제와 트랜잭션 커밋 사이의 미세한 찰나까지 완벽히 보호합니다.
- **재시도**: Redisson의 무거운 구독 방식 대신, Lettuce의 SETNX를 활용한 **지수 백오프(Exponential Backoff)** 기반 스핀락을 직접 구현하여 효율적인 재시도를 수행합니다.

## 3. 구현 시나리오 매트릭스

### 시나리오 A: Lettuce 분산락 적용 (상품 예약 - TradeFacade)

| 단계 | 이서연 (Thread1) | 최우진 (Thread2) | Redis 상태 | DB 상태 (Product) |
|------|-----------------|-----------------|------------|---------|
| T=0 | `lock(key="lock:product:42")` 시도 | `lock(key="lock:product:42")` 시도 | - | SALE |
| T=1 | 락 획득 성공 (SETNX = true) | 락 획득 실패 (백오프 후 스핀락 재시도) | key="lock:product:42" 점유 | SALE |
| T=2 | 트랜잭션 시작 (DB SELECT) | (백오프 중지 대기) | - | SALE |
| T=3 | status = SALE 확인 후 RESERVED 업데이트 (Pessimistic Lock 획득) | (재시도 중이나 락을 얻지 못함) | - | RESERVED (Locked) |
| T=4 | 트랜잭션 COMMIT (Pessimistic Lock 해제) | (재시도 중) | - | RESERVED |
| T=5 | `unlock` (Lua Script 락 해제) | 락 획득 시도 성공 -> 트랜잭션 진입 | key 삭제됨 | RESERVED |
| T=6 | 200 응답 반환 | `status == SALE` 검증 -> 통과실패 (ALREADY_RESERVED 예외) | - | RESERVED |

**코드 구현 예시** (`LockService` + `TradeFacade` 패턴):
```java
// LockService.java — SETNX 방식 분산락
public <T> T executeWithLock(String key, Supplier<T> supplier) {
    String value = UUID.randomUUID().toString();
    // 스핀락: maxRetry, 지수 백오프 제어하며 재시도
    long currentBackoff = initialBackoff;
    while (retryCount < maxRetry) {
        Boolean acquired = redisLockRepository.lock(key, value, duration);
        if (Boolean.TRUE.equals(acquired)) {
            try { return supplier.get(); } 
            finally { redisLockRepository.unlock(key, value); } // Lua 스크립트로 안전 해제
        }
        Thread.sleep(currentBackoff);
        currentBackoff = Math.min(currentBackoff * 2, maxBackoff); // 지수 백오프
        retryCount++;
    }
    throw new ConflictException(ErrorCode.LOCK_TIMEOUT);
}

// TradeFacade.java — 락 담당 (@Transactional 없음)
@Component
public class TradeFacade {
    private final LockService lockService;
    private final TradeService tradeService;

    // 락 획득 후 트랜잭션 서비스 실행 (분산락에서 매우 중요한 AOP 분리 패턴)
    public TradeReserveResponse reserveProduct(Long productId, Long buyerId) {
        String lockKey = "lock:product:" + productId;
        return lockService.executeWithLock(lockKey, () -> tradeService.reserveProduct(productId, buyerId));
    }
}
```

> **Facade 분리 설계의 당위성**: 같은 서비스 클래스 내에 `@Transactional` 어노테이션이 달린 메서드와 락 제어 메서드를 혼합하면, 락의 해제 스코프와 트랜잭션 커밋 스코프 사이의 불일치(트랜잭션 커밋이 일어나기 0.1초 전에 락이 먼저 반환되는 등)로 인한 Lost Update 이슈가 재발할 수 있습니다. `Facade` 클래스를 분리하여 Lock 범위를 확실하게 Transaction 범위보다 넓게(`Lock` 획득 -> `Transaction` 시작 -> `Transaction` 종료 -> `Lock` 해제) 확보했습니다.

### 시나리오 B: 상품 찜하기 (Like) 동시성 제어

상품의 `likeCount` 역시 여러 사용자가 동시에 찜하기/취소를 누를 경우 숫자가 맞지 않는 갱신 손실(Lost Update) 문제가 발생할 수 있습니다. 예약과 동일한 `LockService`를 재사용하여 안전성을 높였습니다.

| 로직 단계 | 설명 |
|------|------|
| 1. Facade 진입 | `LikeFacade.createLike()` 진입 후 대상 productId 기준으로 Redis 분산락 획득 (`LockService`) |
| 2. DB 진입 | 락을 점유한 1개의 스레드만 `LikeService.createLike()` (Transaction 내부) 진입 |
| 3. 관계 저장 | LikeRepository 저장소에서 Like 엔티티 활성화 또는 생성 후 저장 |
| 4. Count 증가| `product.incrementLikeCount()` 호출을 통해 상태 업데이트 |
| 5. DB 커밋 | 트랜잭션이 성공적으로 수행되며 DB 변경 사항이 반영됨. |
| 6. 락 반환 | Redis 키를 삭제(`unlock`)하여 대기 중인 다음 쓰레드로 순번을 넘김. 찜 수 누락 방지! |

---

### 시나리오 C: Redis TTL(Time To Live) 만료에 따른 엣지 케이스 안전성

락의 데드락(무한 점유)을 막기 위해 모든 락에는 `LockExpirationSeconds` TTL(현재 3초)이 부여됩니다.

**상황**: Thread1이 락을 획득했으나 시스템의 엄청난 GC(Garbage Collection)나 외부 호출 지연 등으로 DB 처리가 3초를 넘어가는 경우.

1. **TTL 만료**: Thread1의 `key`가 Redis 단에서 자동 만료 삭제.
2. **동시성 위반 우려**: Thread2가 락을 획득하고 진입. 하지만 시스템은 **애플리케이션의 상태 검증(ProductEnum.SALE 확인)**을 진행하기 때문에, Thread1이 Commit하기 전이라면 Thread2가 락을 획득했어도 최종 처리가 완료된 시점을 기준으로 보면 방어가 어느정도 이루어집니다 (다만, 순수 분산락 100% 한계 케이스로 아주 드물게 Lost Update가 발생할 가능성이 존재함, 본 시스템 수용 범위 내).
3. **Lua Script Unlock 방어**: Thread1이 비즈니스 작업 후 지연되게 `unlock`을 시도하더라도, Thread2가 세팅해둔 `key`를 삭제해버리는 버그를 막기 위해 **Lua 커스텀 스크립트**("본인의 UUID(`value`)와 일치할 때만 `del` 수행")가 적용되어 타 쓰레드의 진행 중인 락을 가로채 훼손하는 문제를 완벽히 차단했습니다.

### 시나리오 D: 동일 사용자 중복 클릭 방지 (멱등성)

**상황**: 사용자가 모바일에서 통신 지연 상황 중 [예약하기] 버튼 2번 다닥 클릭.

| 단계 | 요청 1 (T=0) | 요청 2 (T=10ms) | 현상 및 상태 |
|---|---|---|---|
| 1 | 락 획득 (lock:product:42) | 스핀락 백오프 대기 진입 (LockService) | 락이 잠겼으므로 대기. DB 부하 0 |
| 2 | `status == SALE` 통과, RESERVED로 갱신 | 대기중 ... | 이 시점에 Product 통과 완료 |
| 3 | 트랜잭션 커밋 / 락 해제 | 락 획득하여 트랜잭션 시작 (Service 진입) | - |
| 4 | 클라이언트에 200 정상 반환 | **status가 이미 RESERVED이므로 BusinessException 발생** | 중복 등록(Trade Entity 2개 생성) 완벽 차단 |

프론트엔드 레벨에서의 Double Submit 방지 버튼 비활성화(disabled)가 동반되는 것이 좋습니다.

## 4. 거래 상태 전이 — 동시성 제어 범위

```text
[상태 전이와 락 적용 범위]

SALE ─────────────────────────────────▶ RESERVED (동시 발생 가능성 매우 높음)
     ← Lettuce 분산락(Redis) 적용 구간 →

락 대상 키: "lock:product:{productId}"
동작 방식: TradeFacade를 통해 락 선점. LockService를 이용.

이유별 락 필요성:
- SALE → RESERVED : N명(구매자들) 동시 시도 가능 → 락 필수 ✅
- RESERVED → TRADING : 거래가 매칭된 시점, 판매자/본 구매자 단독 액션 → 일반 트랜잭션 충분
- TRADING → DONE   : 상태 종결, 경쟁 희박 → 일반 트랜잭션 충분

∴ Lettuce 분산락은 다수결 경합이 치열한 SALE → RESERVED 전이 시점 및 누구나 동시에 가능성 높은 **찜하기(likeCount)** 기능에만 한정적으로 적용하여 불필요한 시스템의 성능 저하를 방지 중입니다.
```

## 5. 테스트 시나리오 및 검증 (`TradeConcurrencyTest`)

동시성에 대한 신뢰성을 검증하기 위해 JUnit `CountDownLatch` 및 다중 스레드 생성(`ExecutorService`)을 활용한 테스트 코드를 유지하고 있습니다.

```java
@Test
@DisplayName("동시에 100명이 예약 요청할 경우 단 1명만 성공해야 한다 (분산락 적용)")
void reserveProduct_concurrency() throws InterruptedException {
    int threadCount = 100;
    
    // ... 100명의 테스트용 유저 생성 (생략)

    ExecutorService executorService = Executors.newFixedThreadPool(32);
    CountDownLatch latch = new CountDownLatch(threadCount);
    AtomicInteger successCount = new AtomicInteger();
    AtomicInteger failCount = new AtomicInteger();

    // 100개의 스레드발 동시포격 예약 요청
    for (int i = 0; i < threadCount; i++) {
        Long buyerId = buyerIds.get(i);
        executorService.submit(() -> {
            try {
                // Service 대신 Facade의 락 로직을 호출하여 분산락 제어 확인
                tradeFacade.reserveProduct(productId, buyerId);
                successCount.incrementAndGet();
            } catch (Exception e) {
                failCount.incrementAndGet(); // 예외 발생! (ALREADY_RESERVED 또는 LOCK_TIMEOUT)
            } finally {
                latch.countDown();
            }
        });
    }

    latch.await(10, TimeUnit.SECONDS);

    // 단 1개의 요청만 성공하고 99개는 실패해야 정합성이 맞는 것!
    assertThat(successCount.get()).isEqualTo(1);
    assertThat(failCount.get()).isEqualTo(99);

    // 실제 생성된 Trade 엔티티 개수 검증 (단 1개만 생성)
    assertThat(tradeRepository.count()).isEqualTo(1);
}
```

(Like의 경우에도 `LikeConcurrencyTest.java`에서 동일한 방식으로 100번 호출 시 `likeCount`가 100이 되는지 테스트를 진행하여 검증합니다.)

## 6. 모니터링 포인트 및 락 설정값 (`application.yml`)

`LockService.java` 내부에서 사용되는 변수들입니다.

| 설정 변수 | 기본값 | 이유 및 목적 |
|----------|-------|-------------|
| `lock.retry.max` | 30회 | Lettuce를 활용한 스핀락 방식이므로 무한 재시도를 막아 Redis 부하를 제한하기 위함 |
| `lock.retry.initial-backoff-ms` | 50ms | 락 획득 실패 시 초기 대기 시간 |
| `lock.retry.max-backoff-ms` | 1000ms | 백오프 지수 증가 시 최대 한계 대기 시간 (Thread.sleep). 네트워크 지연 방어 효과 |
| `lock.expiration-seconds` | 3초 | 쓰레드가 락을 잡고 예기치 못하게 죽었을 때 서비스 전체의 무한 교착(Deadlock)을 방지하기 위한 최대 TTL |

> **타임아웃 시뮬레이션 파악**:
> 특정 사용자가 대기 시간이 길어져 결국 30회의 재시도(`maxRetry`)를 모두 허탕치게 될 경우 대략 2초+ 가량이 소요되며, 결국 최종 실패 처리로 간주하고 `ErrorCode.LOCK_TIMEOUT` 커스텀 예외를 발생시켜 클라이언트에 빠른 피드백을 주도록 견고하게 설계되었습니다.
