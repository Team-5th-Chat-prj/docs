# 04. 기능 명세서

> **버전**: v1.0

## 1. 회원 인증

### 1-1. 회원가입

**요청 데이터**: 이메일, 비밀번호, 닉네임

**처리 로직**:
1. 이메일 형식 검증 (RFC 5322)
2. 비밀번호 정책 검증 (8자 이상, 영문+숫자 조합)
3. 이메일 중복 조회 (`SELECT 1 FROM member WHERE email = ?`)
4. BCrypt(strength=10) 비밀번호 해시
5. Member 엔티티 저장
6. 201 Created 반환

**실패 케이스**:
- 이메일 중복 → 409 Conflict `{ "code": "DUPLICATE_EMAIL" }`
- 유효성 오류 → 400 Bad Request

---

### 1-2. 로그인 & JWT 발급

**JWT 구조**:

| 토큰 | 만료 | 저장 위치 | 용도 |
|------|------|-----------|------|
| AccessToken | 15분 | 클라이언트 메모리 | API 인증 헤더 |
| RefreshToken | 7일 | Redis | AccessToken 재발급 |

**AccessToken Payload**:
```json
{
  "sub": "memberId",
  "email": "user@example.com",
  "nickname": "닉네임",
  "iat": 1700000000,
  "exp": 1700000900
}
```

**Redis RefreshToken 키 규칙**: `refresh:{memberId}` → TTL 7일

**처리 로직**:
1. 이메일로 회원 조회
2. BCrypt 비교
3. AccessToken 생성 (HMAC-SHA256)
4. RefreshToken 생성 (UUID)
5. Redis SET `refresh:{memberId}` RefreshToken EX 604800
6. 응답 반환

---

### 1-3. 토큰 재발급

**요청**: `Authorization: Bearer {RefreshToken}`

**처리 로직**:
1. RefreshToken에서 memberId 추출
2. Redis GET `refresh:{memberId}` → 저장값과 비교
3. 일치 시 새 AccessToken 발급
4. (선택) RefreshToken 롤링 갱신

## 2. 상품 관리

### 2-1. 상품 등록

**필드 제약**:

| 필드 | 타입 | 제약 |
|------|------|------|
| title | String | 2~40자, NotBlank |
| description | String | 최대 1000자 |
| price | Integer | 100~99,999,999 |
| categoryId | Long | 소분류 ID |
| status | Enum | 기본값 SALE |

**이미지 처리** (도전 기능 범위 외 — 간단 처리):
- 초기 구현: URL 문자열로 저장 (S3 연동은 Out of Scope)
- 테스트 시 Multipart → 로컬 저장 또는 base64

---

### 2-2. 상품 목록 조회

**페이지네이션**: Cursor 기반 (createdAt + id 조합)

> **왜 Cursor?** Offset 방식은 데이터 삽입/삭제 시 중복/누락 발생. 무한 스크롤 UX에 적합.

**정렬 기본값**: 최신 등록순 (createdAt DESC, id DESC)

**응답 필드**: id, title, price, thumbnailUrl, status, createdAt, likeCount

---

### 2-3. 상품 검색 (Caffeine + QueryDSL)

**검색 조건**:

| 파라미터 | 설명 | 필수 |
|----------|------|------|
| keyword | 제목 LIKE 검색 | 선택 |
| categoryId | 소분류 ID | 선택 |
| status | 판매 상태 필터 | 선택 (기본: SALE) |

**QueryDSL 동적 쿼리 예시**:
```java
BooleanBuilder builder = new BooleanBuilder();
if (keyword != null) builder.and(product.title.containsIgnoreCase(keyword));
if (categoryId != null) builder.and(product.category.id.eq(categoryId));
if (status != null) builder.and(product.status.eq(status));
```

**인기 검색어 Caffeine 캐싱**:

| 항목 | 값 |
|------|----|
| 캐시 이름 | `popularKeywords` |
| TTL | 10분 |
| Max Size | 1 (단일 키 — top10 리스트) |
| 갱신 트리거 | 스케줄러 10분 주기 |
| 갱신 방법 | DB에서 `keyword_log` 집계 후 캐시 PUT |

```
검색 요청 → Caffeine 캐시 HIT? → 반환
                       ↓ MISS
              DB에서 TOP 10 집계 → 캐시 PUT → 반환
```

## 3. 거래 상태 관리 ⭐

### 3-1. 상태 전이 규칙

| 현재 상태 | 이벤트 | 다음 상태 | 처리 주체 |
|-----------|--------|-----------|-----------|
| SALE | 예약 요청 + 수락 | RESERVED | 구매자 요청 → 판매자 수락 OR 판매자가 바로 예약 확정 |
| RESERVED | 거래 시작 | TRADING | 판매자 |
| RESERVED | 예약 취소 | SALE | 구매자 또는 판매자 |
| TRADING | 거래 완료 | SOLD | 판매자 |
| TRADING | 거래 취소 | SALE | 판매자 |
| SOLD | 리뷰 작성 완료 | REVIEWED | 구매자 |

> **설계 결정**: 예약 요청을 판매자가 수락하는 2단계로 구현한다.
> 이유: 당근마켓 UX와 동일. 판매자가 거래 상대를 선택할 수 있어야 한다.

---

### 3-2. 동시성 제어 — 예약 처리 상세

**문제**: 동시에 N명이 예약 요청 → 1명만 성공해야 함

**해결 전략**:

| 전략 | 장점 | 단점 | 적용 여부 |
|------|------|------|-----------|
| 낙관적 락 (version) | 충돌 적을 때 성능 좋음 | 재시도 로직 복잡, 충돌 많으면 성능 저하 | ❌ |
| 비관적 락 (SELECT FOR UPDATE) | 구현 단순, 정합성 보장 | 단일 DB 의존, 락 경합 시 지연 | ✅ 기본 |
| Redisson 분산락 | 단일 DB 의존 없음, 분산 환경 대응 | Redis 의존성 추가 | ✅ 도전 기능 |

**구현 계획**:
- **필수**: 비관적 락 (`@Lock(LockModeType.PESSIMISTIC_WRITE)`)
- **도전**: 비관적 락 위에 Redisson 분산락 추가 (이중 방어)

**비관적 락 흐름**:
```
요청 수신
  → @Transactional 시작
  → SELECT product WHERE id=? FOR UPDATE  ← DB 레벨 락
  → status == SALE 확인
  → status = RESERVED 업데이트
  → transaction commit  ← 락 자동 해제
  → 다른 요청: 락 해제 후 status=RESERVED 확인 → 즉시 실패 반환
```

**Redisson 분산락 추가 흐름** (도전 기능):
```
요청 수신
  → tryLock(key="product:{id}", waitTime=0, leaseTime=3s)
  → 락 획득 실패 → 즉시 409 반환 (DB 접근 없음)
  → 락 획득 성공 → 비관적 락 로직 실행
  → finally: unlock
```

> waitTime=0인 이유: 대기하면 사용자는 "응답 없음" 경험. 즉시 실패가 UX에 좋다.

## 4. 채팅 (WebSocket/STOMP)

### 4-1. 연결 흐름

```
클라이언트 → WS 핸드셰이크 (/ws-chat)
           → STOMP CONNECT (JWT 헤더 포함)
           → 서버: JWT 검증 → 세션 등록
           → STOMP SUBSCRIBE /user/queue/messages (개인 채널)
           → STOMP SUBSCRIBE /topic/room.{chatRoomId} (채팅방 채널)
```

### 4-2. 메시지 전송 흐름

```
클라이언트 SEND → /app/chat.sendMessage
서버:
  1. MessageDTO 역직렬화
  2. chatRoom 참여자 검증
  3. Message 엔티티 저장 (MySQL)
  4. /topic/room.{chatRoomId} 로 브로드캐스트
```

### 4-3. 메시지 저장 정책

- **저장소**: MySQL (chat_message 테이블)
- **이유**: Redis Pub/Sub은 영속성 없음. 채팅 내역 재조회 필요
- **페이지네이션**: Cursor 기반 (messageId DESC)
- **연결 끊김 처리**: 재연결 시 마지막 수신 messageId 이후 REST API로 조회

## 5. 찜하기

- 회원당 상품 1개 → 1개 찜 (중복 불가)
- UNIQUE KEY (member_id, product_id) — DB 레벨 중복 방지
- 찜 수는 product 테이블의 `like_count` 컬럼으로 캐싱 (정합성 허용 오차 ±1)
