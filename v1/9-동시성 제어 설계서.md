# 09. 동시성 제어 설계서 ⭐

> **버전**: v1.0
> **대상 기능**: 상품 예약 요청 (1명만 예약 가능)

---

## 1. 문제 정의

### 상황

```
T=0ms  이서연: POST /products/42/reserve  ──┐
T=1ms  최우진: POST /products/42/reserve  ──┤──▶ 동시 도달
T=2ms  박준서: POST /products/42/reserve  ──┘

기대: 이서연만 성공, 최우진·박준서는 즉시 실패
위험: 3명 모두 product.status = "SALE" 읽고 모두 RESERVED로 변경 시도
결과(제어 없을 때): 3명 모두 예약 성공 → 데이터 정합성 파괴
```

### 왜 단순 `if (status == SALE)` 체크로 충분하지 않은가?

```
Thread1: SELECT status → SALE  (읽기)
Thread2: SELECT status → SALE  (읽기, Thread1이 아직 UPDATE 전)
Thread1: UPDATE status = RESERVED  (쓰기)
Thread2: UPDATE status = RESERVED  (쓰기, 이미 RESERVED인데도 덮어씀)
→ Lost Update 문제 발생
```

## 2. 동시성 제어 전략 비교

| 전략 | 메커니즘 | 장점 | 단점 | 적합 케이스 |
|------|----------|------|------|-------------|
| **낙관적 락** | @Version 필드 + 재시도 | 충돌 적을 때 성능 우수 | 재시도 로직 복잡, 충돌 많으면 성능 저하 | 충돌 빈도 낮은 경우 |
| **비관적 락** | SELECT FOR UPDATE | 구현 단순, 정합성 확실 | 락 대기 지연, 단일 DB 의존 | ✅ 본 프로젝트 (필수) |
| **Redisson 분산락** | Redis SetNX + TTL | 분산 환경 대응, DB 부하 감소 | Redis 의존성 추가, 복잡도 증가 | ✅ 본 프로젝트 (도전) |
| **DB Unique 제약** | UNIQUE(product_id) on trade | 최후 방어선 | 에러 처리 복잡, 사용자 경험 불량 | 보조 수단 |

**선택**: 비관적 락(필수) + Redisson(도전) 이중 방어

## 3. 시나리오 매트릭스

### 시나리오 A: 비관적 락만 적용 (필수 구현)

| 단계 | 이서연 (Thread1) | 최우진 (Thread2) | DB 상태 |
|------|-----------------|-----------------|---------|
| T=0 | 트랜잭션 시작 | 트랜잭션 시작 | product.status = SALE |
| T=1 | `SELECT ... FOR UPDATE` 락 획득 | `SELECT ... FOR UPDATE` 대기 (블로킹) | 락 보유: Thread1 |
| T=2 | status = SALE 확인 | (대기 중) | - |
| T=3 | `UPDATE status = RESERVED` | (대기 중) | status = RESERVED |
| T=4 | 트랜잭션 COMMIT, 락 해제 | 락 획득 | status = RESERVED |
| T=5 | 예약 성공 응답 반환 | status = RESERVED 확인 → 실패 | - |
| T=6 | - | 트랜잭션 ROLLBACK | status = RESERVED |
| T=7 | - | 409 응답 반환 | - |

**결과**: 정합성 보장 ✅

**코드 예시**:
```java
// ProductRepository.java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdWithLock(@Param("id") Long id);

// TradeService.java
@Transactional
public TradeResponse reserve(Long productId, Long buyerId) {
    Product product = productRepository.findByIdWithLock(productId)
        .orElseThrow(() -> new NotFoundException("상품을 찾을 수 없습니다."));

    if (product.getStatus() != ProductStatus.SALE) {
        throw new ConflictException("ALREADY_RESERVED", "이미 예약된 상품입니다.");
    }

    product.changeStatus(ProductStatus.RESERVED);
    Trade trade = Trade.create(product, buyerId);
    tradeRepository.save(trade);
    return TradeResponse.from(trade);
}
```

### 시나리오 B: Redisson 분산락 + 비관적 락 (도전 기능)

| 단계 | 이서연 (Thread1) | 최우진 (Thread2) | Redis 상태 | DB 상태 |
|------|-----------------|-----------------|------------|---------|
| T=0 | `tryLock(key="product:42", wait=0, lease=3s)` | `tryLock(key="product:42", wait=0, lease=3s)` | - | SALE |
| T=1 | 락 획득 성공 | 락 획득 실패 (즉시) | key="product:42" SET | SALE |
| T=2 | (비관적 락 로직 실행) | 409 응답 반환 | - | SALE |
| T=3 | SELECT FOR UPDATE → RESERVED | - | - | RESERVED |
| T=4 | COMMIT | - | - | RESERVED |
| T=5 | unlock (Redis key 삭제) | - | key 삭제 | RESERVED |
| T=6 | 200 응답 반환 | - | - | - |

**최우진은 DB에 전혀 도달하지 않음 → DB 부하 감소**

**코드 예시**:
```java
// TradeService.java (Redisson 적용)
@RequiredArgsConstructor
public class TradeService {
    private final RedissonClient redissonClient;
    private final ProductRepository productRepository;

    public TradeResponse reserve(Long productId, Long buyerId) {
        String lockKey = "product:" + productId;
        RLock lock = redissonClient.getLock(lockKey);

        boolean acquired = false;
        try {
            acquired = lock.tryLock(0, 3, TimeUnit.SECONDS); // wait=0: 즉시 실패
            if (!acquired) {
                throw new ConflictException("ALREADY_RESERVED", "이미 예약된 상품입니다.");
            }
            return reserveWithPessimisticLock(productId, buyerId); // 내부에서 @Transactional
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("락 획득 중 인터럽트 발생", e);
        } finally {
            if (acquired && lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }

    @Transactional
    protected TradeResponse reserveWithPessimisticLock(Long productId, Long buyerId) {
        Product product = productRepository.findByIdWithLock(productId)
            .orElseThrow(() -> new NotFoundException("상품을 찾을 수 없습니다."));

        if (product.getStatus() != ProductStatus.SALE) {
            throw new ConflictException("ALREADY_RESERVED", "이미 예약된 상품입니다.");
        }

        product.changeStatus(ProductStatus.RESERVED);
        Trade trade = Trade.create(product, buyerId);
        return TradeResponse.from(tradeRepository.save(trade));
    }
}
```

> **왜 `@Transactional`을 내부 메서드에 두는가?**
> Redisson 락은 트랜잭션 밖에서 잡아야 한다.
> 트랜잭션 COMMIT 전에 락을 해제하면 다른 스레드가 아직 반영 안 된 DB 상태를 읽는 위험이 있기 때문에,
> 반드시 COMMIT 후 finally에서 unlock한다.

### 시나리오 C: 엣지 케이스 — Redis 장애 시

| 상황 | 동작 | 이유 |
|------|------|------|
| Redis 연결 불가 | 비관적 락만으로 처리 (fallback) | Redisson 락 실패 시 예외 처리 → 비관적 락 로직으로 직접 진행 |
| Redis 레이턴시 급증 | leaseTime=3s → 3초 후 자동 해제 | Watchdog 비활성화 (tryLock 사용 시 자동 연장 없음) |
| 서버 크래시 (락 미해제) | leaseTime=3s 후 자동 만료 | Redis TTL 보장 |

## 4. 거래 상태 전이 — 동시성 제어 범위

```
[상태 전이와 락 적용 범위]

SALE ──────────────────────────▶ RESERVED
      ← 이 구간만 동시성 제어 필요 →

이유:
- SALE → RESERVED: 여러 명이 동시에 시도 가능 (락 필수)
- RESERVED → TRADING: 판매자 단독 작업 (중복 없음)
- TRADING → SOLD: 판매자 단독 작업 (중복 없음)
- SOLD → REVIEWED: 구매자 단독 작업 (중복 없음)

∴ 비관적 락 + Redisson은 SALE → RESERVED 전이에만 적용
```

## 5. 테스트 시나리오

### 단위 테스트

```java
@Test
@DisplayName("동시에 5명이 예약 시도 시 1명만 성공해야 한다")
void concurrentReserveTest() throws InterruptedException {
    // given
    int threadCount = 5;
    ExecutorService executor = Executors.newFixedThreadPool(threadCount);
    CountDownLatch latch = new CountDownLatch(threadCount);
    AtomicInteger successCount = new AtomicInteger(0);
    AtomicInteger failCount = new AtomicInteger(0);

    // when
    for (int i = 0; i < threadCount; i++) {
        final long buyerId = i + 1;
        executor.submit(() -> {
            try {
                tradeService.reserve(PRODUCT_ID, buyerId);
                successCount.incrementAndGet();
            } catch (ConflictException e) {
                failCount.incrementAndGet();
            } finally {
                latch.countDown();
            }
        });
    }
    latch.await(10, TimeUnit.SECONDS);

    // then
    assertThat(successCount.get()).isEqualTo(1);
    assertThat(failCount.get()).isEqualTo(4);

    Product product = productRepository.findById(PRODUCT_ID).orElseThrow();
    assertThat(product.getStatus()).isEqualTo(ProductStatus.RESERVED);
}
```
