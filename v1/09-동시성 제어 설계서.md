# 09. 동시성 제어 설계서 ⭐

> **버전**: v1.2 (예약 모델 즉시 RESERVED 정합성 확인, 락 대상 명시)
> **대상 기능**: 상품 예약 요청 (1명만 예약 가능)
>
> **예약 모델 정책**: 구매자가 [예약하기] 클릭 시 **판매자 승인 없이 즉시 RESERVED** (전 문서 공통)
> **락 대상**: `PRODUCT` 테이블 — 상품 상태(SALE→RESERVED) 변경의 원천. TRADE는 락 획득 성공 후 생성됨.

---

## 1. 문제 정의

### 상황

```
T=0ms  이서연: POST /products/42/reserve  ──┐
T=1ms  최우진: POST /products/42/reserve  ──┤──▶ 동시 도달
T=2ms  박준서: POST /products/42/reserve  ──┘

기대: 이서연만 성공, 최우진·박준서는 즉시 실패
위험: 3명 모두 product.status = "SALE" 읽고 모두 RESERVED로 변경 시도
결과(제어 없을 때): 3명 모두 예약 성공 → 데이터 정합성 파괴
```

### 왜 단순 `if (status == SALE)` 체크로 충분하지 않은가?

```
Thread1: SELECT status → SALE  (읽기)
Thread2: SELECT status → SALE  (읽기, Thread1이 아직 UPDATE 전)
Thread1: UPDATE status = RESERVED  (쓰기)
Thread2: UPDATE status = RESERVED  (쓰기, 이미 RESERVED인데도 덮어씀)
→ Lost Update 문제 발생
```

## 2. 동시성 제어 전략 비교

| 전략 | 메커니즘 | 장점 | 단점 | 적합 케이스 |
|------|----------|------|------|-------------|
| **낙관적 락** | @Version 필드 + 재시도 | 충돌 적을 때 성능 우수 | 재시도 로직 복잡, 충돌 많으면 성능 저하 | 충돌 빈도 낮은 경우 |
| **비관적 락** | SELECT FOR UPDATE | 구현 단순, 정합성 확실 | 락 대기 지연, 단일 DB 의존 | ✅ 본 프로젝트 (필수) |
| **Redisson 분산락** | Redis SetNX + TTL | 분산 환경 대응, DB 부하 감소 | Redis 의존성 추가, 복잡도 증가 | ✅ 본 프로젝트 (도전) |
| **DB Unique 제약** | UNIQUE(product_id) on trade | 최후 방어선 | 에러 처리 복잡, 사용자 경험 불량 | 보조 수단 |

**선택**: 비관적 락(필수) + Redisson(도전) 이중 방어

## 3. 시나리오 매트릭스

### 시나리오 A: 비관적 락만 적용 (필수 구현)

| 단계 | 이서연 (Thread1) | 최우진 (Thread2) | DB 상태 |
|------|-----------------|-----------------|---------|
| T=0 | 트랜잭션 시작 | 트랜잭션 시작 | product.status = SALE |
| T=1 | `SELECT ... FOR UPDATE` 락 획득 | `SELECT ... FOR UPDATE` 대기 (블로킹) | 락 보유: Thread1 |
| T=2 | status = SALE 확인 | (대기 중) | - |
| T=3 | `UPDATE status = RESERVED` | (대기 중) | status = RESERVED |
| T=4 | 트랜잭션 COMMIT, 락 해제 | 락 획득 | status = RESERVED |
| T=5 | 예약 성공 응답 반환 | status = RESERVED 확인 → 실패 | - |
| T=6 | - | 트랜잭션 ROLLBACK | status = RESERVED |
| T=7 | - | 409 응답 반환 | - |

**결과**: 정합성 보장 ✅

**코드 예시**:
```java
// ProductRepository.java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdWithLock(@Param("id") Long id);

// TradeServiceImpl.java
@Transactional
public TradeResponse reserveWithPessimisticLock(Long productId, Long buyerId) {
    Product product = productRepository.findByIdWithLock(productId)
        .orElseThrow(() -> new NotFoundException("상품을 찾을 수 없습니다."));

    if (product.getStatus() != ProductStatus.SALE) {
        throw new ConflictException("ALREADY_RESERVED", "이미 예약된 상품입니다.");
    }

    product.changeStatus(ProductStatus.RESERVED);
    Trade trade = Trade.create(product, buyerId);
    tradeRepository.save(trade);
    return TradeResponse.from(trade);
}
```

### 시나리오 B: Redisson 분산락 + 비관적 락 (도전 기능)

| 단계 | 이서연 (Thread1) | 최우진 (Thread2) | Redis 상태 | DB 상태 |
|------|-----------------|-----------------|------------|---------|
| T=0 | `tryLock(key="lock:product:42", wait=0, lease=3s)` | `tryLock(key="lock:product:42", wait=0, lease=3s)` | - | SALE |
| T=1 | 락 획득 성공 | 락 획득 실패 (즉시) | key="lock:product:42" SET | SALE |
| T=2 | (비관적 락 로직 실행) | 409 응답 반환 | - | SALE |
| T=3 | SELECT FOR UPDATE → RESERVED | - | - | RESERVED |
| T=4 | COMMIT | - | - | RESERVED |
| T=5 | unlock (Redis key 삭제) | - | key 삭제 | RESERVED |
| T=6 | 200 응답 반환 | - | - | - |

**최우진은 DB에 전혀 도달하지 않음 → DB 부하 감소**

**코드 예시** (TradeLockFacade + TradeServiceImpl 분리 패턴):
```java
// TradeLockFacade.java — 락 담당 (@Component, @Transactional 없음)
@Component
@RequiredArgsConstructor
public class TradeLockFacade {
    private final RedissonClient redissonClient;
    private final TradeService tradeService; // 별도 빈 — 프록시를 통해 호출

    public TradeResponse reserve(Long productId, Long buyerId) {
        String lockKey = "lock:product:" + productId;
        RLock lock = redissonClient.getLock(lockKey);

        boolean acquired = false;
        try {
            acquired = lock.tryLock(0, 3, TimeUnit.SECONDS); // wait=0: 즉시 실패
            if (!acquired) {
                throw new ConflictException("ALREADY_RESERVED", "이미 예약된 상품입니다.");
            }
            return tradeService.reserveWithPessimisticLock(productId, buyerId); // 외부 빈 호출 → AOP 프록시 정상 작동
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("락 획득 중 인터럽트 발생", e);
        } finally {
            if (acquired && lock.isHeldByCurrentThread()) {
                lock.unlock(); // 트랜잭션 COMMIT 이후 실행 보장
            }
        }
    }
}

// TradeServiceImpl.java — DB 처리 담당 (@Transactional 적용)
@Service
@RequiredArgsConstructor
public class TradeServiceImpl implements TradeService {
    private final ProductRepository productRepository;
    private final TradeRepository tradeRepository;

    @Override
    @Transactional
    public TradeResponse reserveWithPessimisticLock(Long productId, Long buyerId) {
        Product product = productRepository.findByIdWithLock(productId)
            .orElseThrow(() -> new NotFoundException("상품을 찾을 수 없습니다."));

        if (product.getStatus() != ProductStatus.SALE) {
            throw new ConflictException("ALREADY_RESERVED", "이미 예약된 상품입니다.");
        }

        product.changeStatus(ProductStatus.RESERVED);
        Trade trade = Trade.create(product, buyerId);
        return TradeResponse.from(tradeRepository.save(trade));
    }
}
```

> **Facade 분리 이유**: 같은 클래스 내에서 `this.reserveWithPessimisticLock()`을 호출하면 Spring AOP 프록시를 우회하여 `@Transactional`이 적용되지 않는다. `TradeLockFacade`가 트랜잭션 없이 락을 잡고, `TradeServiceImpl`(별도 빈)을 호출해야 Spring이 `@Transactional` 프록시를 통해 트랜잭션을 시작한다.
> COMMIT 이후 `finally` 블록에서 `unlock`이 실행되므로, 다른 스레드가 아직 반영 안 된 DB 상태를 읽는 위험도 없다.

### 시나리오 C: 엣지 케이스 — Redis 장애 시

| 상황 | 동작 | 이유 |
|------|------|------|
| Redis 연결 불가 | 비관적 락만으로 처리 (fallback) | Redisson 락 실패 시 예외 처리 → 비관적 락 로직으로 직접 진행 |
| Redis 레이턴시 급증 | leaseTime=3s → 3초 후 자동 해제 | Watchdog 비활성화 (tryLock 사용 시 자동 연장 없음) |
| 서버 크래시 (락 미해제) | leaseTime=3s 후 자동 만료 | Redis TTL 보장 |

### 시나리오 D: 비관적 락 대기 중 innodb_lock_wait_timeout 초과

**상황**: Thread1이 락을 보유한 채 처리 지연 → Thread2가 MySQL 기본 대기시간(50초) 초과

| 단계 | 이서연 (Thread1) | 최우진 (Thread2) | DB 상태 |
|------|-----------------|-----------------|---------|
| T=0 | SELECT FOR UPDATE 락 획득 | SELECT FOR UPDATE 대기 시작 | SALE, 락 보유: Thread1 |
| T=1~49s | (처리 지연 — 외부 API 호출 등) | 대기 중 | - |
| T=50s | (아직 처리 중) | `LockTimeoutException` 발생 | - |
| T=50s | - | 트랜잭션 ROLLBACK | SALE (Thread1 아직 미완료) |
| T=51s | COMMIT | - | RESERVED |

**발생 예외**: `javax.persistence.LockTimeoutException` → Spring이 `QueryTimeoutException`으로 래핑

**처리 전략**:
```java
// application.yml — 락 대기 타임아웃 명시적 설정 (기본 50s → 3s로 단축)
spring:
  jpa:
    properties:
      javax.persistence.lock.timeout: 3000  # 밀리초

// GlobalExceptionHandler.java
@ExceptionHandler(LockTimeoutException.class)
public ResponseEntity<ErrorResponse> handleLockTimeout(LockTimeoutException e) {
    return ResponseEntity.status(409)
        .body(new ErrorResponse("LOCK_TIMEOUT", "잠시 후 다시 시도해주세요."));
}
```

**핵심 설계 결정**: 기본 50초 대기는 사용자 경험 최악. `lock.timeout=3000ms`로 단축하고 즉시 친절한 에러 반환.

### 시나리오 E: Redisson leaseTime 만료 중 처리 지연 경쟁

**상황**: tryLock 성공 후 leaseTime(3초) 내에 트랜잭션이 완료되지 않을 경우

| 단계 | 이서연 (Thread1) | 최우진 (Thread2) | Redis | DB |
|------|-----------------|-----------------|-------|----|
| T=0 | tryLock 성공 (leaseTime=3s) | tryLock 실패 → 409 반환 | key SET, TTL=3s | SALE |
| T=1~2s | DB 처리 중 (지연 발생) | - | TTL 감소 중 | - |
| T=3s | **leaseTime 만료 → Redis key 자동 삭제** | - | key 없음 | 미완료 |
| T=3s | (트랜잭션은 아직 진행 중) | tryLock 성공 가능 상태 | - | SALE |
| T=3.1s | - | 새 요청: tryLock 성공 | key SET | SALE |
| T=3.2s | - | SELECT FOR UPDATE 대기 | - | Thread1이 락 보유 |
| T=4s | COMMIT (비관적 락 보호로 정합성 유지됨) | 락 획득 → status=RESERVED 확인 → 409 | key 삭제 | RESERVED |

**결과**: leaseTime이 만료됐어도 **비관적 락이 최후 방어선으로 정합성 보장** ✅

**왜 Watchdog(자동 연장)을 사용하지 않는가**:
```
tryLock() → leaseTime 고정, 자동 연장 없음
lock()    → Watchdog 활성화, 30초마다 자동 연장

∴ tryLock 사용 시 leaseTime은 반드시 실제 처리 시간보다 충분히 길어야 한다.
  현재 설정 3초는 DB 트랜잭션이 통상 100ms~500ms임을 감안한 값.
  외부 API 호출이 추가되면 leaseTime을 재검토해야 한다.
```

### 시나리오 F: 동일 사용자 중복 클릭 (멱등성)

**상황**: 이서연이 네트워크 지연으로 [예약하기] 버튼을 2번 연속 클릭

| 단계 | 이서연 요청1 | 이서연 요청2 | DB 상태 |
|------|-------------|-------------|---------|
| T=0 | POST /products/42/reserve | POST /products/42/reserve (50ms 후 재클릭) | SALE |
| T=1 | Redisson 락 획득 | Redisson 락 획득 실패 → 즉시 409 반환 | SALE |
| T=2 | SELECT FOR UPDATE — status=SALE 확인 | - | SALE |
| T=3 | UPDATE status=RESERVED, Trade 생성 | - | RESERVED |
| T=4 | COMMIT | - | RESERVED |
| T=5 | 200 응답 | 409 응답 이미 전달됨 | RESERVED |

**Redisson 락으로 1차 차단되지만, 락 해제 후 재시도 시 추가 방어 필요**:

```java
// 동일 구매자의 중복 예약 방지 — DB 레벨 가드 추가
@Transactional
protected TradeResponse reserveWithPessimisticLock(Long productId, Long buyerId) {
    Product product = productRepository.findByIdWithLock(productId)
        .orElseThrow(() -> new NotFoundException("상품을 찾을 수 없습니다."));

    if (product.getStatus() != ProductStatus.SALE) {
        throw new ConflictException("ALREADY_RESERVED", "이미 예약된 상품입니다.");
    }

    // ✅ 추가: 동일 구매자가 이미 이 상품에 활성 거래가 있는지 확인
    boolean alreadyReservedByMe = tradeRepository
        .existsByProductIdAndBuyerIdAndStatusNot(productId, buyerId, TradeStatus.CANCELLED);
    if (alreadyReservedByMe) {
        throw new ConflictException("ALREADY_RESERVED", "이미 예약한 상품입니다.");
    }

    product.changeStatus(ProductStatus.RESERVED);
    Trade trade = Trade.create(product, buyerId);
    return TradeResponse.from(tradeRepository.save(trade));
}
```

**클라이언트 측 보완**: 예약 버튼 클릭 후 응답 전까지 비활성화(disabled) 처리 → 프론트엔드 가이드에 명시 필요

### 시나리오 G: 예약 취소와 새 예약의 경쟁 (상태 롤백 경쟁)

**상황**: 기존 예약자(이서연)가 취소하는 동시에 새 구매자(최우진)가 예약 시도

| 단계 | 이서연 (취소 요청) | 최우진 (예약 요청) | DB 상태 |
|------|------------------|------------------|---------|
| T=0 | PATCH /trades/99/status { "status": "SALE" } | POST /products/42/reserve | RESERVED |
| T=1 | SELECT FOR UPDATE (TRADE + PRODUCT) | Redisson tryLock → 락 획득 시도 | RESERVED |
| T=1 | 락 획득 성공 | Redisson 락 획득 실패 (product:42 key 존재) → 409 | RESERVED |
| T=2 | product.status = SALE, trade.status = CANCELLED | - | SALE (취소 완료 중) |
| T=3 | COMMIT | - | SALE |
| T=4 | 락 해제 | **최우진: 재시도 없음** (wait=0) | SALE |

**문제**: 최우진은 SALE 상태가 됐음에도 409를 받고 재시도 기회가 없음

**설계 결정 — 재시도 정책**:

```
방안 1: 클라이언트 재시도 유도
  → 409 응답에 retryAfter 힌트 포함
  { "code": "ALREADY_RESERVED", "retryAfter": 1 }
  → 프론트에서 1초 후 상태 재조회 후 예약 버튼 활성화

방안 2: 상품 상세 페이지 polling
  → 취소 이벤트 발생 시 상품 status가 SALE로 바뀌면
    구독 중인 클라이언트에게 SSE/WebSocket으로 상태 변경 통보
    (현재 스코프 외 — 향후 확장)

∴ 현재 구현: 방안 1 (retryAfter 힌트) 채택
```

**응답 예시 수정** (06_api_spec.md 연동):
```json
{
  "code": "ALREADY_RESERVED",
  "message": "이미 예약된 상품입니다. 잠시 후 다시 확인해보세요.",
  "retryAfter": 1
}
```

## 4. 거래 상태 전이 — 동시성 제어 범위

```
[상태 전이와 락 적용 범위]

SALE ─────────────────────────────────▶ RESERVED (즉시, 판매자 승인 없음)
     ← 비관적 락 + Redisson 적용 구간 →

락 대상 테이블: PRODUCT (SELECT ... FOR UPDATE)
락 키 (Redisson): "lock:product:{productId}"
TRADE 생성: 락 보유 중, 트랜잭션 내에서 PRODUCT 상태 변경 직후 생성

이유별 락 필요성:
- SALE → RESERVED : N명 동시 시도 가능 → 락 필수 ✅
- RESERVED → TRADING : 판매자 단독 액션, 경쟁 없음 → 락 불필요
- TRADING → SOLD   : 판매자 단독 액션, 경쟁 없음 → 락 불필요
- SOLD → REVIEWED  : 구매자 단독 액션, 경쟁 없음 → 락 불필요

∴ 비관적 락 + Redisson은 SALE → RESERVED 전이에만 적용
```

## 5. 테스트 시나리오

### 단위 테스트

```java
@Test
@DisplayName("동시에 5명이 예약 시도 시 1명만 성공해야 한다")
void concurrentReserveTest() throws InterruptedException {
    // given
    int threadCount = 5;
    ExecutorService executor = Executors.newFixedThreadPool(threadCount);
    CountDownLatch latch = new CountDownLatch(threadCount);
    AtomicInteger successCount = new AtomicInteger(0);
    AtomicInteger failCount = new AtomicInteger(0);

    // when
    for (int i = 0; i < threadCount; i++) {
        final long buyerId = i + 1;
        executor.submit(() -> {
            try {
                tradeService.reserve(PRODUCT_ID, buyerId);
                successCount.incrementAndGet();
            } catch (ConflictException e) {
                failCount.incrementAndGet();
            } finally {
                latch.countDown();
            }
        });
    }
    latch.await(10, TimeUnit.SECONDS);

    // then
    assertThat(successCount.get()).isEqualTo(1);
    assertThat(failCount.get()).isEqualTo(4);

    Product product = productRepository.findById(PRODUCT_ID).orElseThrow();
    assertThat(product.getStatus()).isEqualTo(ProductStatus.RESERVED);
}
```

## 6. 모니터링 포인트

| 항목 | 확인 방법 | 임계값 |
|------|-----------|--------|
| 락 획득 실패율 | 409 응답 비율 로그 | 일반적으로 OK |
| DB 락 대기 타임아웃 | `LockTimeoutException` 로그 | 빈번 발생 시 leaseTime 재검토 |
| Redisson leaseTime 만료 | leaseTime 내 트랜잭션 완료 여부 | 초과 시 leaseTime 값 증가 검토 |
| 중복 예약 시도율 | `ALREADY_RESERVED` 에러 빈도 | UI 버튼 disabled 처리 확인 |
| 데드락 발생 | MySQL error log | 1회 발생 시 즉시 조사 |

### 설정값 기준표

| 설정 | 현재값 | 이유 |
|------|--------|------|
| `javax.persistence.lock.timeout` | 3,000ms | 기본 50s → UX 개선 |
| Redisson `leaseTime` | 3s | 통상 트랜잭션 500ms 기준 여유 6배 |
| Redisson `waitTime` | 0s | 대기 없이 즉시 실패 → UX 개선 |
