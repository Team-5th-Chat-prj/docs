## 2. 유스케이스 명세

### UC01. 회원가입

| 항목 | 내용 |
|------|------|
| 액터 | 비회원 |
| 사전 조건 | 이메일이 중복되지 않아야 함 |
| 기본 흐름 | 1. 이메일, 비밀번호, 닉네임 입력 → 2. 이메일 중복 확인 → 3. 비밀번호 해시(BCrypt) → 4. 회원 저장 → 5. 성공 응답 |
| 예외 흐름 | E1. 이메일 중복 → 409 Conflict 반환 |
| 사후 조건 | 회원 레코드 생성, 로그인 가능 상태 |

---

### UC02. 로그인

| 항목 | 내용 |
|------|------|
| 액터 | 비회원 |
| 사전 조건 | 가입된 이메일 |
| 기본 흐름 | 1. 이메일/비밀번호 입력 → 2. BCrypt 비교 → 3. AccessToken(15분) + RefreshToken(7일) 발급 → 4. RefreshToken Redis 저장 |
| 예외 흐름 | E1. 비밀번호 불일치 → 401 Unauthorized |
| 사후 조건 | JWT 쌍 반환 |

---

| 기본 흐름 | 1. RefreshToken 전송 → 2. Redis에서 검증 → 3. 새 AccessToken 발급 |
| 예외 흐름 | E1. RefreshToken 만료/불일치 → 401, 재로그인 유도 |

### UC16. 찜하기 / 찜 취소

| 항목 | 내용 |
|------|------|

### UC30. 채팅방 생성

| 항목 | 내용 |
|------|------|
| 액터 | 회원(구매자), 시스템 |
| 사전 조건 | 로그인 상태, 예약 요청 성공 |
| 기본 흐름 | **예약 요청(UC20) 성공 시 자동 생성** — 별도 API 호출 없음. TradeService 내에서 ChatRoom 엔티티를 함께 저장하고 chatRoomId를 응답에 포함 |
| 설계 결정 | 채팅방 독립 생성 API를 두지 않은 이유: 채팅은 반드시 거래와 연결되어야 하며, 거래 없는 채팅방은 허용하지 않음 |
| 예외 흐름 | 예약 실패 시 채팅방도 생성되지 않음 (트랜잭션 원자성) |
| 사후 조건 | CHAT_ROOM 레코드 생성, 응답에 chatRoomId 포함 |

### UC40. 인기 검색어 조회

| 항목 | 내용 |
|------|------|
| 액터 | 비회원, 회원, 시스템(스케줄러) |
| 사전 조건 | 없음 |

## 3. 거래 상태 전이 다이어그램 (PlantUML)

```plantuml
@startuml
skinparam state {
  BackgroundColor LightBlue
  BorderColor DarkBlue
}

[*] --> SALE : 상품 등록

SALE --> RESERVED : 예약 요청 + 판매자 수락\n[Redisson 분산락]
SALE --> DELETED : 판매자 삭제

RESERVED --> SALE : 예약 취소 (판매자/구매자)
RESERVED --> TRADING : 판매자 거래 시작 확인

TRADING --> SOLD : 판매자 거래 완료 처리
TRADING --> SALE : 거래 취소

SOLD --> REVIEWED : 구매자 리뷰 작성

REVIEWED --> [*]
DELETED --> [*]

note right of RESERVED
  이 상태에서 다른 구매자의
  예약 요청은 모두 거절됨
  (동시성 제어 핵심 구간)
end note
@enduml
```
